# homework4_data_base
# Практичне завдання №4: Система управління бібліотекою

## Огляд проєкту
Метою цього проєкту є проєктування, реалізація та оптимізація операційної бази даних PostgreSQL для **Системи управління бібліотекою**. Система керує читальними залами, жанрами, книгами, користувачами бібліотеки, їхніми профілями, історією видачі книг, а також автоматичним нарахуванням штрафів за прострочене повернення.

Цілісне рішення повністю задовольняє основні вимоги (20 балів) та реалізує всі додаткові завдання для отримання бонусних балів (+3 бали).

База даних складається з 7 таблиць, які розроблені для забезпечення реальних бібліотечних операцій та моделювання всіх необхідних типів зв'язків:

1. **`Reading_Halls`**: Містить інформацію про конкретні навчальні або читальні зали.
2. **`Genres`**: Зберігає категорії (жанри) книг.
3. **`Books`**: Каталог книг бібліотеки, прив'язаний до жанрів та конкретних читальних залів.
4. **`Users`**: Реєстраційні дані членів бібліотеки (читачів).
5. **`User_Profiles`**: Детальні персональні дані користувачів (паспортні дані, адреса тощо).
6. **`Borrow_Log`**: Головна операційна таблиця, де зберігається історія та активні записи про видачу книг (цільова таблиця для генерації 500 000+ рядків).
7. **`Fines`**: Відстежує фінансові штрафи за несвоєчасно повернуті книги.

### Реалізовані типи зв'язків:
* **1:1 (Один-до-одного)**: `Users` ||--|| `User_Profiles` (Кожен користувач має рівно один унікальний профіль із паспортними даними).
* **1:Many (Один-до-багатьох)**: 
  * `Genres` 1--o{ `Books` (Один жанр містить багато книг).
  * `Reading_Halls` 1--o{ `Books` (В одному залі може розміщуватися багато книг).
  * `Borrow_Log` 1--o{ `Fines` (Один запис про видачу може згенерувати один або кілька штрафів у разі затримки).
* **Many:Many (Багато-до-багатьох)**: `Users` та `Books` пов'язані через проміжну таблицю `Borrow_Log` (Один користувач може брати багато різних книг протягом певного часу, а одну й ту саму книгу в різний час можуть брати різні користувачі).

## Опис файлів та архітектури проєкту

Репозиторій проєкту містить такі файли:

1. **`create_tables.sql`**: DDL-скрипт, який видаляє наявні структури (якщо вони були) і створює всі 7 таблиць із комплексними обмеженнями (`PRIMARY KEY`, `FOREIGN KEY`, `NOT NULL`, `UNIQUE` та `CHECK`).

2. **`main.py`**: Python-скрипт, який використовує драйвер `psycopg2` та бібліотеку `Faker`. Він підключається до бази даних і наповнює таблиці понад **500 000 операційних записів** у таблиці `Borrow_Log`, використовуючи пакетну вставку (batch insertion) для максимальної продуктивності.

3. **`create_view.sql`**: Створює аналітичне представлення бази даних (`v_debtors`), яке об'єднує дані з кількох таблиць (`Users`, `Books`, `Borrow_Log`), щоб виводити актуальний список поточних боржників бібліотеки.

4. **`create_user.sql`**: Налаштування системи доступу на основі ролей (RBAC). Створює 3 різних користувачів/ролей бази даних із чітко розмежованими правами доступу (`GRANT SELECT`, `INSERT`, `UPDATE` для конкретних таблиць).

5. **Збережена процедура та Тригер (включені в рішення)**:
   * **Процедура**: `register_new_borrow` обробляє безпечні транзакції видачі книг, перевіряючи наявність вільних екземплярів перед зменшенням залишку.
   * **Тригер**: Автоматично вираховує період прострочення під час повернення книги та динамічно вносить запис у таблицю `Fines`, якщо було зафіксовано порушення термінів.

## Оптимізація та тестування продуктивності

Для демонстрації оптимізації продуктивності на великому обсязі даних (500 000+ рядків у `Borrow_Log`) використовується команда `EXPLAIN ANALYZE`.

### До оптимізації (Послідовне сканування / Sequential Scan)
Пошукові запити за неіндексованими зовнішніми ключами або атрибутами (наприклад, пошук записів видачі за ID користувача чи його прізвищем) змушують PostgreSQL виконувати повне сканування таблиці (`Seq Scan`), що призводить до високого часу виконання запиту.

### Після оптимізації (Сканування індексу / Index Scan)
